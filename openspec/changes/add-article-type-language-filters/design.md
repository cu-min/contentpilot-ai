# 设计：文章库增加类型与语言筛选

## 上下文

文章列表接口已经接受并校验 `type` 和 `language` 参数，前端的 `ArticleQuery` 类型及选项数据也已具备对应字段。当前缺口仅是文章库页面未把两个字段传入查询。

本次变更只补齐前端筛选交互。没有 API、数据库、外部平台、权限或发布状态机变更，因此不需要修改 `docs/contracts/`。

## 目标与不做事项

**目标：**

- 让运营人员在文章库按文章类型和语言单独或组合筛选。
- 复用现有接口、前端类型和下拉选项，保持改动范围最小。
- 让重置和翻页行为与现有关键词、状态筛选保持一致。

**不做事项：**

- 不增加后端接口、数据库字段或新的查询参数。
- 不增加排序、筛选条件持久化或多条件标签展示。
- 不修改发布任务及人工发布边界。

## 设计决策

### 决策一：只修改文章库前端页面

在 `frontend/src/pages/Article/index.tsx` 的现有查询表单中增加“文章类型”和“语言”下拉框，并在加载列表时把表单中的 `type` 与 `language` 一并传给 `getArticles`。

原因：后端 `ArticleQueryRequest`、服务层过滤逻辑和前端 `ArticleQuery` 已支持这两个参数，修改后端会增加无价值风险。

备选方案：新增专用筛选接口。未采用，因为会重复现有能力并增加维护成本。

### 决策二：复用统一的选项与表单状态

复用 `frontend/src/types/article.ts` 中的 `articleTypeOptions` 与 `articleLanguageOptions`，并继续由现有 Form 管理所有筛选条件。

原因：可避免页面自行维护枚举和标签，保证创建、编辑和查询页面的选项一致。

### 决策三：分页沿用现有查询方法

分页仍调用同一个 `loadArticles` 方法，并从表单读取当前条件。

原因：当前关键词与状态筛选已使用这一方式，能够自然保留新增条件。

## 风险与取舍

- [筛选条件与接口字段不一致] → 复用现有 `ArticleQuery` 类型和选项常量，不在页面手写枚举。
- [重置后仍使用旧页码] → 重置操作固定重新加载第一页。
- [页面宽度不足] → 复用现有内联表单与可清空下拉框；如窄屏出现换行，保持表单可用，不增加响应式重构。

## 发布与回滚

这是纯前端兼容性变更，可随前端版本发布。若发现问题，回滚本页面新增的两个筛选控件即可；后端和数据库无需回滚。

## 共享文件与负责人

- 前端负责人：`frontend/src/pages/Article/index.tsx`。
- 不涉及共享热点文件，不需要后端或架构变更。
