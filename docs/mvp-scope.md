# MVP 范围说明

## 当前 MVP 最小目标

当前 MVP 只保留一条可以端到端验证的主链路：

```text
产品配置
↓
AI 生成文章
↓
平台适配稿
↓
平台账号配置
↓
创建发布任务
↓
手动执行发布任务
↓
记录成功链接或失败原因
```

阶段 8 的重点不是接入真实平台，而是先验证系统内部发布任务执行链路是否清晰、可扩展、可回滚。

## 当前保留模块

- 产品配置
- AI 文章生成
- 文章库
- 多平台内容适配
- 平台账号配置
- 发布任务草稿流转
- 发布执行框架
- MockPublisher 模拟发布
- 用户、角色与 JWT 鉴权
- 首页概览

## 当前暂停或砍掉模块

以下能力暂不纳入当前 MVP：

- 追踪链接模块
- 真实数据看板
- 平台阅读量、点赞、收藏、评论统计
- 自动定时调度
- 复杂失败重试
- Redis
- SaaS 多租户
- 自动浏览
- 自动评论
- 操作日志
- 通知提醒
- 多平台同时真实发布
- 自动读取所有平台发布链接
- ECharts 复杂图表

## 追踪链接和真实数据看板为什么暂不做

追踪链接和真实数据看板依赖稳定的发布结果、平台链接和访问数据。当前系统还没有接入真实平台发布器，提前建设追踪链路会扩大表结构、接口和前端页面复杂度，也会增加后续返工风险。

当前 MVP 先把内容生成、平台适配、账号配置、发布任务和执行结果跑通。真实平台发布稳定后，再补充追踪链接和数据看板更稳。

## 为什么先做发布执行框架

不同平台的发布方式差异很大：

- 微信公众号优先官方 API 和草稿。
- 知乎更适合浏览器自动化加人工确认。
- CSDN 偏浏览器自动填充或草稿。
- 掘金可先试点非官方草稿接口，再用浏览器兜底。

阶段 8 先抽象 `PlatformPublisher`、`PublishContext`、`PublishResult` 和 `PublisherRegistry`，让 `PublishTaskService` 不直接写死具体平台实现。后续接入单个平台时，只需要新增对应 Publisher，并复用现有任务状态流转。

## MockPublisher 的作用

`MockPublisher` 只用于阶段 8：

- 不调用任何真实平台接口。
- 不读取真实 Cookie、Token、API Key。
- 默认模拟发布成功。
- 成功时返回 `https://mock.publish/{platform}/{taskId}`。
- 当标题、摘要、正文、标签、关键词或账号备注包含 `mock-fail` 时模拟失败。

它的价值是验证：

- `PENDING -> RUNNING -> SUCCESS`
- `PENDING -> RUNNING -> FAILED`
- `publish_url` 写入
- `error_message` 写入
- 前端执行按钮、loading、状态 Tag、结果展示

## 后续最小路线

- 阶段 8：发布执行框架 + MockPublisher。
- 阶段 8.5：掘金 / CSDN 发布方案调研与试点计划，不写真实发布代码。
- 阶段 9：掘金草稿发布试点。
- 阶段 10：CSDN 自动填充试点，可选。
- 阶段 11：MVP 联调与演示优化。

真实数据看板、追踪链接、通知提醒、复杂重试、多平台同时发布等能力放到后续扩展阶段。
