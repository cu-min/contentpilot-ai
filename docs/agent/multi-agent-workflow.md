# 多 Agent 开发流程

## 目标

通过稳定上下文、明确所有权、前后端并行和自动验证缩短交付时间，同时避免为了流程而增加不必要的状态、表格和沟通。

## 角色

### Team Lead

- 与用户沟通需求和优先级。
- 决定启用哪些角色、任务如何拆分以及依赖顺序。
- 维护简洁的任务列表并跟踪阻塞。
- 汇总结果，不写业务代码，不代替其他角色完成工作。

### 架构 Agent

- 设计必要的业务流程、数据变化、接口和状态。
- 冻结前后端共同依赖的契约。
- 指定文件所有权和共享文件负责人。
- 审查开发结果是否符合契约和现有架构。
- 执行无冲突集成；出现冲突时退回原负责人处理。

### 后端 Agent

- 在后端 Worktree 实现后端功能。
- 维护后端测试并运行完整后端验证。
- 不静默修改已冻结的前后端契约。

### 前端 Agent

- 在前端 Worktree 实现页面、交互、状态和 API 接入。
- 维护前端测试并运行完整前端验证。
- 不自行猜测后端尚未定义的字段或状态。

### QA Agent

- 在集成结果上执行构建、测试、契约检查和业务冒烟。
- 输出可复现的问题和证据，并执行回归验证。
- 不直接修改前后端业务代码；问题交回对应负责人。

## 运行流程

### 1. 明确任务

Team Lead 至少明确：

- 要解决的问题和预期结果
- 基本验收条件
- 明确不做的内容
- 是否涉及真实平台、账号或外部服务

小任务可以直接进入实现；涉及前后端、数据库、公共状态或外部平台的任务必须先经过架构 Agent。

### 2. 冻结必要契约

架构 Agent 只设计本任务真正需要的内容，通常包括：

- API 路径、请求、响应和错误行为
- 数据库字段或迁移
- 前后端共享枚举和状态
- 页面关键交互和异常状态
- 文件所有权和共享修改点

契约可以直接写在任务文档中，不要求每个任务创建完整 OpenAPI。复杂或长期接口再维护到 `docs/contracts/`。

### 3. 并行实现

前端、后端从同一个契约提交开始，各自在专属 Worktree 工作。

- 技术问题发送给架构 Agent。
- 进度、阻塞和完成消息发送给 Team Lead。
- 影响另一端的变更必须主动通知。
- 如果契约确实需要改变，说明原因和影响，等待架构 Agent 更新后继续。

### 4. 自验和交接

开发 Agent 完成后，先运行自己负责范围的验证。交接信息使用自然语言即可，但必须包含：

- 当前任务和状态
- 主要实现
- 修改范围
- 验证命令与结果
- 必要的风险、阻塞或契约偏差
- 分支或提交

### 5. 审查和集成

架构 Agent 审查契约一致性、修改范围和共享文件，然后将合格结果集成。无须为每个小改动写长审查报告；有问题时给出具体文件、原因和退回对象。

### 6. QA 验证

QA 在集成后执行：

- 后端测试
- 前端构建和测试
- 必要的 Docker 启动检查
- 核心业务流程冒烟
- 本任务相关的异常和边界场景

真实平台发布默认不进入无人值守测试，需要用户批准并准备本地凭据。

### 7. 交付

Team Lead 向用户说明：

- 完成了什么
- 自动验证结果
- 是否经过真实环境验证
- 剩余限制和风险
- 后续建议（仅在确有必要时）

## 效率原则

- 能并行才拆分，强依赖或同文件任务保持串行。
- 文档只写会影响实现和验收的决策。
- 小任务减少角色，大任务再启用完整团队。
- 尽早暴露阻塞，不等待所有成员结束后才报告。
- 保留证据和结果，不保留冗长过程记录。
