# AI内容营销系统 产品设计文档（MVP版）

## 1. 项目概述

### 1.1 项目名称
AI内容营销系统

### 1.2 产品定位
AI内容营销系统是一个面向公司内部运营使用的内容营销自动化平台。系统围绕单一公司产品进行内容生产、平台适配、发布任务流转和发布结果记录，帮助运营人员提高内容生产与分发效率。

初版定位为内部简易工具，不做复杂 SaaS 能力；但在平台适配、发布器接口、AI模型接口等技术设计上保留后续扩展空间。

### 1.3 核心目标
- 根据产品配置和推广主题，自动生成中文或英文营销文章。
- 针对微信公众号、知乎、CSDN、掘金生成不同平台版本。
- 支持立即发布和定时发布计划的任务管理，当前 MVP 已接入微信公众号、掘金、CSDN、知乎的真实发布或自动化试点链路。
- 通过统一 Publisher 架构记录发布成功链接或失败原因，为后续更多真实平台发布器预留扩展。
- 追踪链接和真实数据看板后续扩展，暂不纳入当前 MVP。
- 提供文章库、平台账号、发布任务和首页概览。操作日志已取消，暂不纳入初版。

### 1.4 初版边界
| 类型 | 初版是否支持 | 说明 |
|---|---:|---|
| 公司内部使用 | 支持 | 不做多租户和计费 |
| 多产品配置 | 支持 | 可维护多个产品；生成时可选择产品，也可按主题生成通用文章 |
| 国内平台发布 | 支持 | 公众号、知乎、CSDN、掘金 |
| 发布执行 | 支持 | 当前 MVP 支持立即发布和定时发布时间记录，均由运营人员手动执行 |
| 真实平台自动发布 | 部分支持 | 微信公众号官方 API、掘金非官方接口试点、CSDN/知乎浏览器自动化已接入 |
| 追踪链接 | 暂不支持 | 后续扩展，不纳入当前 MVP |
| 真实数据看板 | 暂不支持 | 当前仅保留简单首页概览 |
| 手动录入平台数据 | 不支持 | 避免背离自动化方向 |
| 图片/素材库 | 不支持 | 初版以文字内容为主 |
| 复杂文章版本管理 | 不支持 | 不做历史版本对比和回滚 |
| Redis | 不使用 | 初版降低复杂度 |
| SaaS能力 | 不做 | 后续再考虑 |
| 自动浏览/自动评论 | 不做 | 仅作为后续扩展评估项 |

## 2. 用户角色与权限

### 2.1 角色划分
| 角色 | 说明 | 权限 |
|---|---|---|
| 管理员 | 公司内部管理人员 | 比运营人员多用户管理和权限管理 |
| 运营人员 | 内容运营人员 | 生成文章、管理文章、创建发布任务、执行发布任务 |

### 2.2 权限原则
初版采用简单角色权限模型，不做复杂按钮级权限。管理员和运营人员共享大部分业务能力，管理员额外拥有用户管理能力。

### 2.3 用户相关功能
- 登录
- 注册或管理员创建用户
- 修改用户状态
- 设置用户角色
- JWT鉴权
- 退出登录

## 3. 产品配置模块

### 3.1 模块定位
产品配置用于为 AI 生成内容提供可选的基础上下文。系统支持维护多个产品；生成文章时可选择一个产品，也可不选产品生成通用文章。未选择时系统不会使用或猜测其他产品的品牌与事实信息。

### 3.2 配置字段
| 字段 | 说明 |
|---|---|
| 产品名称 | 公司当前推广产品名称 |
| 产品简介 | 产品核心介绍 |
| 官网链接 | 默认推广跳转目标 |
| 核心功能 | 产品主要功能点 |
| 目标用户 | 目标客户群体 |
| 产品优势 | 与竞品相比的优势 |
| 品牌语气 | 专业、清晰、技术向等 |
| 禁用词/敏感词 | AI生成内容时需要规避的词 |

### 3.3 使用方式
产品配置会作为所选产品的 AI 文章生成和平台适配提示词上下文，减少运营人员重复输入背景信息。

## 5. 多平台内容适配模块

### 5.1 初版平台
- 微信公众号
- 知乎
- CSDN
- 掘金

### 5.2 平台适配原则
同一个主题不会直接复制到所有平台，而是根据不同平台生成不同版本。平台适配不是扩写，而是先理解原文核心观点，再按平台阅读习惯进行删减、重组和改写。能减则减，删除不适合当前平台的重复背景、过长铺垫、过强硬广、不必要技术细节和复杂长段落。

| 平台 | 保留重点 | 删减重点 | 结构与语气 |
|---|---|---|---|
| 微信公众号 | 核心痛点、产品价值、场景化案例、清晰小标题、适度品牌表达 | 过长技术细节、大段代码、复杂表格、太多行业背景铺垫、太硬销售话术 | 移动端短段落，开头有场景或痛点，结尾可温和引导；专业、易读、有传播感 |
| 知乎 | 问题背景、逻辑推导、对比分析、真实痛点、客观建议 | 明显营销口吻、过多产品自夸、口号式表达、直接销售转化话术、空泛结论 | 问题型或观点型标题，分点论证，产品自然出现；理性、客观、有观点 |
| CSDN | 技术背景、操作步骤、功能说明、使用场景、示例和配置说明 | 品牌故事、运营化表达、情绪化标题、空泛价值描述、与技术实践无关段落 | 实用标题、步骤编号、代码块或配置示例；技术、实用、清晰 |
| 掘金 | 实践经验、开发者痛点、工具效率、场景总结、方法论 | 官方品牌宣传、企业营销口吻、过长背景铺垫、纯概念解释、无实践价值段落 | 技术价值感标题，从开发者痛点切入，偏经验总结；轻量、实用、有分享感 |

### 5.3 文章保存策略
系统保存不同平台的发布稿，但不做复杂文章版本管理。也就是说，不做历史版本回溯、版本对比、修改记录和版本恢复。

### 6.1 模块功能
- 查看文章列表
- 查看文章详情
- 查看不同平台发布稿
- 查看发布状态
- 查看发布任务状态和发布结果
- 归档文章

### 6.2 文章状态
| 状态 | 说明 |
|---|---|
| 草稿 | AI生成后尚未进入发布流程 |
| 待发布 | 已创建发布任务 |
| 已发布 | 发布成功并读取到文章链接 |

## 7. 发布执行模块

### 7.1 模块定位
发布执行是系统 MVP 主链路的末端能力。系统支持创建发布任务，并在提交时选择立即发布或定时发布。两类任务均在 `PENDING` 状态由运营人员手动执行；定时任务仅保存计划时间，当前 MVP 不轮询或自动执行。当前微信公众号 Publisher 走官方 API，掘金 Publisher 走非官方接口试点，CSDN 和知乎 Publisher 走 Playwright 浏览器自动化。成功记录发布链接，失败记录可读原因。

### 7.2 发布方式分级

#### Level 1：官方接口发布
后续优先使用平台官方接口完成发布。适用于有稳定开放能力的平台。

#### Level 2：浏览器自动化发布
后续当平台没有稳定开放接口时，系统可通过浏览器自动化打开平台页面，自动填写标题、正文、标签等信息，并尽量完成提交。

#### Level 3：自动填充 + 人工确认兜底
如果平台出现验证码、登录失效、页面结构变化或风控限制，系统可降级为自动填充页面，由运营人员最后确认发布。

### 7.3 自动发布边界
系统不提供以下能力：
- 绕过验证码
- 绕过登录验证
- 规避平台风控
- 批量刷量
- 模拟真人行为对抗平台限制

### 7.4 发布任务流程
```text
创建发布任务
↓
配置平台与发布时间
↓
提交为待执行
↓
立即发布：运营人员手动触发
定时发布：到达计划时间后由运营人员手动触发
↓
按平台 Publisher 执行发布
↓
发布成功：记录正式文章链接
发布失败：记录失败状态和错误信息
```

### 7.5 发布任务状态
| 状态 | 说明 |
|---|---|
| DRAFT | 草稿，可编辑 |
| PENDING | 待执行 |
| RUNNING | 执行中 |
| SUCCESS | 发布成功 |
| FAILED | 发布失败 |
| CANCELLED | 已取消 |

### 7.6 失败处理
初版失败后不自动重试，直接记录错误信息和失败状态。

### 7.7 发布链接获取
当前发布成功必须记录 Publisher 返回的发布链接；无法确认成功链接时，任务记为 `FAILED` 并记录原因。

## 8. 平台账号配置模块

### 8.1 账号策略
初版采用公司统一平台账号，不做每个运营人员独立绑定账号。

### 8.2 配置内容
- 平台名称
- 账号标识
- 登录态配置
- Token/Cookie/授权信息
- 启用状态
- 最后更新时间

### 8.3 安全要求
平台密钥、Token、Cookie、浏览器用户目录等敏感信息不能写死在前端或代码中，应存储在后端配置中，并避免日志明文输出。掘金真实发布依赖本地平台账号配置中的 Cookie、分类和标签信息；CSDN 和知乎浏览器自动化依赖本地 Chrome for Testing 用户目录复用登录态。

## 9. 追踪链接模块（后续扩展）

### 9.1 模块定位
追踪链接可作为后续数据观测能力，用于统计访问和跳转效果。当前 MVP 暂不实现追踪链接生成、跳转、点击记录和相关接口。

### 9.2 工作流程
```text
系统生成追踪链接
↓
文章中插入追踪链接
↓
用户点击追踪链接
↓
系统记录点击数据
↓
跳转到真实产品官网
```

### 9.3 每平台独立链接
同一篇文章在不同平台发布时，需要生成不同追踪链接。

示例：
- 公众号追踪链接
- 知乎追踪链接
- CSDN追踪链接
- 掘金追踪链接

### 9.4 记录数据
初版不记录访问者 IP 和设备信息，只记录低敏数据：
- 文章ID
- 平台
- 追踪链接ID
- 访问时间
- 点击次数
- 跳转目标链接

## 10. 数据看板模块（后续扩展）

### 10.1 数据来源
当前 MVP 只保留简单首页概览，不建设真实数据看板。后续数据看板可基于以下数据来源扩展：

- 追踪链接访问日志
- 发布任务记录
- 文章状态数据

### 10.2 核心指标
- 文章总数
- 已发布文章数
- 各平台发布数量
- 追踪链接访问量
- 官网跳转点击量
- 各平台点击排名
- 文章点击排名
- 最近 7 天访问趋势

### 10.3 平台维度统计
数据看板需要按平台统计，便于运营判断不同平台的推广效果。

当前 MVP 不实现 ECharts 复杂图表、平台阅读量、点赞、收藏、评论统计。

## 11. 操作日志模块（已取消）

用户已决定后续不再将操作日志作为独立模块。初版不实现 operation_log 表、操作日志接口、操作日志菜单和操作日志页面。

系统仍可在发布任务等业务表中保存必要的业务状态和错误信息，但不建设独立审计日志能力。

## 18. 总结

AI内容营销系统初版应聚焦一个闭环：

```text
产品配置 → AI生成 → 平台适配 → 平台账号配置 → 发布任务 → 发布执行结果
```

该闭环先体现 AI 内容生产、平台适配和发布执行链路。当前 MVP 已完成多个真实平台发布试点，但仍不做追踪链接、真实数据看板、自动失败重试、验证码绕过或平台风控规避，为后续数据归因、SaaS 化、AI 互动辅助等能力留出扩展空间。
