# AI内容营销系统 产品设计文档（MVP版）

## 1. 项目概述

### 1.1 项目名称
AI内容营销系统

### 1.2 产品定位
AI内容营销系统是一个面向公司内部运营使用的内容营销自动化平台。系统围绕单一公司产品进行内容生产、平台适配、自动发布、链接追踪和数据观测，帮助运营人员提高内容生产与分发效率。

初版定位为内部简易工具，不做复杂 SaaS 能力；但在平台适配、发布器接口、AI模型接口等技术设计上保留后续扩展空间。

### 1.3 核心目标
- 根据产品配置和推广主题，自动生成中文或英文营销文章。
- 针对微信公众号、知乎、CSDN、掘金生成不同平台版本。
- 支持立即发布和定时自动发布。
- 自动读取发布后的文章链接。
- 为不同平台文章生成独立追踪链接，用于统计访问与跳转效果。
- 提供文章库、发布任务和数据看板。操作日志已取消，暂不纳入初版。

### 1.4 初版边界
| 类型 | 初版是否支持 | 说明 |
|---|---:|---|
| 公司内部使用 | 支持 | 不做多租户和计费 |
| 单产品配置 | 支持 | 只推广一个产品 |
| 国内平台发布 | 支持 | 公众号、知乎、CSDN、掘金 |
| 自动发布 | 支持 | 优先自动化，失败直接报错 |
| 手动录入平台数据 | 不支持 | 避免背离自动化方向 |
| 图片/素材库 | 不支持 | 初版以文字内容为主 |
| 复杂文章版本管理 | 不支持 | 不做历史版本对比和回滚 |
| Redis | 不使用 | 初版降低复杂度 |
| SaaS能力 | 不做 | 后续再考虑 |
| 自动浏览/自动评论 | 不做 | 仅作为后续扩展评估项 |

## 2. 用户角色与权限

### 2.1 角色划分
| 角色 | 说明 | 权限 |
|---|---|---|
| 管理员 | 公司内部管理人员 | 比运营人员多用户管理和权限管理 |
| 运营人员 | 内容运营人员 | 生成文章、管理文章、创建发布任务、查看数据 |

### 2.2 权限原则
初版采用简单角色权限模型，不做复杂按钮级权限。管理员和运营人员共享大部分业务能力，管理员额外拥有用户管理能力。

### 2.3 用户相关功能
- 登录
- 注册或管理员创建用户
- 修改用户状态
- 设置用户角色
- JWT鉴权
- 退出登录

## 3. 产品配置模块

### 3.1 模块定位
产品配置用于为 AI 生成内容提供基础上下文。系统当前只支持单产品，不支持多产品切换。

### 3.2 配置字段
| 字段 | 说明 |
|---|---|
| 产品名称 | 公司当前推广产品名称 |
| 产品简介 | 产品核心介绍 |
| 官网链接 | 默认推广跳转目标 |
| 核心功能 | 产品主要功能点 |
| 目标用户 | 目标客户群体 |
| 产品优势 | 与竞品相比的优势 |
| 品牌语气 | 专业、清晰、技术向等 |
| 禁用词/敏感词 | AI生成内容时需要规避的词 |

### 3.3 使用方式
产品配置会作为 AI 文章生成和平台适配提示词的固定上下文，减少运营人员重复输入背景信息。

### 4.1 模块目标
根据主题、文章类型、语言和产品配置，生成适合营销推广的文章内容。

### 4.2 支持语言
- 国内中文文章
- 国外英文文章

### 4.3 文章类型
| 类型 | 说明 |
|---|---|
| 产品介绍 | 介绍产品能力和价值 |
| 使用教程 | 指导用户完成某类操作 |
| 行业科普 | 围绕行业问题做知识科普 |
| 竞品对比 | 与同类产品或方案进行对比 |
| 解决方案 | 面向具体业务问题提供方案 |

### 4.4 AI输出内容
- 标题
- 摘要

## 5. 多平台内容适配模块

### 5.1 初版平台
- 微信公众号
- 知乎
- CSDN
- 掘金

### 5.2 平台适配原则
同一个主题不会直接复制到所有平台，而是根据不同平台生成不同版本。平台适配不是扩写，而是先理解原文核心观点，再按平台阅读习惯进行删减、重组和改写。能减则减，删除不适合当前平台的重复背景、过长铺垫、过强硬广、不必要技术细节和复杂长段落。

| 平台 | 保留重点 | 删减重点 | 结构与语气 |
|---|---|---|---|
| 微信公众号 | 核心痛点、产品价值、场景化案例、清晰小标题、适度品牌表达 | 过长技术细节、大段代码、复杂表格、太多行业背景铺垫、太硬销售话术 | 移动端短段落，开头有场景或痛点，结尾可温和引导；专业、易读、有传播感 |
| 知乎 | 问题背景、逻辑推导、对比分析、真实痛点、客观建议 | 明显营销口吻、过多产品自夸、口号式表达、直接销售转化话术、空泛结论 | 问题型或观点型标题，分点论证，产品自然出现；理性、客观、有观点 |
| CSDN | 技术背景、操作步骤、功能说明、使用场景、示例和配置说明 | 品牌故事、运营化表达、情绪化标题、空泛价值描述、与技术实践无关段落 | 实用标题、步骤编号、代码块或配置示例；技术、实用、清晰 |
| 掘金 | 实践经验、开发者痛点、工具效率、场景总结、方法论 | 官方品牌宣传、企业营销口吻、过长背景铺垫、纯概念解释、无实践价值段落 | 技术价值感标题，从开发者痛点切入，偏经验总结；轻量、实用、有分享感 |

### 5.3 文章保存策略
系统保存不同平台的发布稿，但不做复杂文章版本管理。也就是说，不做历史版本回溯、版本对比、修改记录和版本恢复。

## 6. 文章库模块

### 6.1 模块功能
- 查看文章列表
- 查看文章详情
- 查看不同平台发布稿
- 查看发布状态
- 查看追踪数据
- 归档文章

### 6.2 文章状态
| 状态 | 说明 |
|---|---|
| 草稿 | AI生成后尚未进入发布流程 |
| 待发布 | 已创建发布任务 |
| 已发布 | 发布成功并读取到文章链接 |
| 发布失败 | 发布任务失败 |

### 6.3 编辑能力
初版使用 Markdown 编辑器 + 预览。编辑能力只做基础查看、简单修改、保存、预览，不重点建设复杂富文本编辑器。

## 7. 自动发布模块

### 7.1 模块定位
自动发布是系统核心能力。系统支持创建发布任务，并按配置时间自动发布到目标平台。

### 7.2 发布方式分级

#### Level 1：官方接口发布
优先使用平台官方接口完成发布。适用于有稳定开放能力的平台。

#### Level 2：浏览器自动化发布
当平台没有稳定开放接口时，系统通过浏览器自动化打开平台页面，自动填写标题、正文、标签等信息，并尽量完成提交。

#### Level 3：自动填充 + 人工确认兜底
如果平台出现验证码、登录失效、页面结构变化或风控限制，系统可降级为自动填充页面，由运营人员最后确认发布。

### 7.3 自动发布边界
系统不提供以下能力：
- 绕过验证码
- 绕过登录验证
- 规避平台风控
- 批量刷量
- 模拟真人行为对抗平台限制

### 7.4 发布任务流程
```text
创建发布任务
↓
配置平台与发布时间
↓
到达发布时间
↓
执行自动发布
↓
发布成功：自动读取文章链接
发布失败：直接报错并记录失败状态和错误信息
```

### 7.5 发布任务状态
| 状态 | 说明 |
|---|---|
| PENDING | 待执行 |
| RUNNING | 执行中 |
| SUCCESS | 发布成功 |
| FAILED | 发布失败 |

### 7.6 失败处理
初版失败后不自动重试，直接记录错误信息和失败状态。

### 7.7 发布链接获取
发布成功后系统自动读取发布后的文章链接。若无法读取链接，任务标记为链接获取失败或发布异常。

## 8. 平台账号配置模块

### 8.1 账号策略
初版采用公司统一平台账号，不做每个运营人员独立绑定账号。

### 8.2 配置内容
- 平台名称
- 账号标识
- 登录态配置
- Token/Cookie/授权信息
- 启用状态
- 最后更新时间

### 8.3 安全要求
平台密钥、Token、Cookie 等敏感信息不能写死在前端或代码中，应存储在后端配置中，并避免日志明文输出。

## 9. 追踪链接模块

### 9.1 模块定位
追踪链接是数据观测核心。系统不强依赖各平台的阅读量、点赞数、收藏数 API，而是通过自有追踪链接统计访问和跳转。

### 9.2 工作流程
```text
系统生成追踪链接
↓
文章中插入追踪链接
↓
用户点击追踪链接
↓
系统记录点击数据
↓
跳转到真实产品官网
```

### 9.3 每平台独立链接
同一篇文章在不同平台发布时，需要生成不同追踪链接。

示例：
- 公众号追踪链接
- 知乎追踪链接
- CSDN追踪链接
- 掘金追踪链接

### 9.4 记录数据
初版不记录访问者 IP 和设备信息，只记录低敏数据：
- 文章ID
- 平台
- 追踪链接ID
- 访问时间
- 点击次数
- 跳转目标链接

## 10. 数据看板模块

### 10.1 数据来源
- 追踪链接访问日志
- 发布任务记录
- 文章状态数据

### 10.2 核心指标
- 文章总数
- 已发布文章数
- 各平台发布数量
- 追踪链接访问量
- 官网跳转点击量
- 各平台点击排名
- 文章点击排名
- 最近 7 天访问趋势

### 10.3 平台维度统计
数据看板需要按平台统计，便于运营判断不同平台的推广效果。

## 11. 操作日志模块（已取消）

用户已决定后续不再将操作日志作为独立模块。初版不实现 operation_log 表、操作日志接口、操作日志菜单和操作日志页面。

系统仍可在发布任务、追踪链接等业务表中保存必要的业务状态和错误信息，但不建设独立审计日志能力。

## 13. 技术架构设计

- MobX

- Markdown Editor

### 13.2 后端技术栈
- Java
- Spring Boot
- Spring Security
- JWT
- MyBatis-Plus
- MySQL
- Spring Scheduling

### 13.3 不使用 Redis
初版不引入 Redis，降低部署和开发复杂度。

### 13.4 浏览器自动化
推荐使用 Playwright。相比 Selenium，Playwright 对现代浏览器自动化支持较好，适合自动打开页面、填写表单、处理基础发布流程。

- 错误提示优化
- 文档整理

## 18. 自动浏览 / 自动评论后续扩展分析

### 18.1 功能定位
自动浏览和自动评论不进入初版，仅作为后续待评估功能。

### 18.2 优点
- 可以辅助发现相关内容和潜在线索。
- 可以提升运营人员互动效率。
- 可以生成评论建议，减少人工构思成本。

### 18.3 风险
- 平台规则限制较多。
- 容易触发风控。
- 账号安全风险高。
- 技术维护成本高。
- 不适合作为初版核心功能。

### 18.4 建议形态
后续不建议做“自动刷浏览、自动评论”，建议升级为“AI互动辅助模块”。

可做能力：
- 发现相关话题文章
- 生成评论建议
- 人工确认后发布
- 评论情绪分析
- 潜在线索识别

## 19. 总结

AI内容营销系统初版应聚焦一个闭环：

```text
产品配置 → AI生成 → 平台适配 → 自动发布 → 追踪链接 → 数据看板
```

该闭环既能体现 AI 内容生产能力，也能体现自动化发布和数据观测价值。初版不追求平台全部打通，而是先把核心架构和最小可运行流程做出来，为后续全平台发布、SaaS化、AI互动辅助等能力留出扩展空间。
