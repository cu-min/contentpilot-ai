# 多平台内容适配规则

## 核心原则

平台适配不是扩写，也不是把原文简单复制到不同平台。系统需要先理解原始文章的核心观点、产品价值、用户痛点、解决方案和转化引导，再根据目标平台进行删减、重组和改写。

本阶段采用“能减则减”的原则：删除不适合当前平台的重复背景、过长铺垫、过强硬广、不必要技术细节、复杂长段落和与平台阅读目标无关的内容。平台稿不需要保留原文所有段落，可以调整标题、结构、段落顺序和表达方式，但不能编造产品功能、案例、数据或平台能力。

AI 输出仍必须是 JSON：

```json
{
  "title": "...",
  "summary": "...",
  "content": "...",
  "tags": ["..."],
  "keywords": ["..."]
}
```

`content` 使用 Markdown。生成过程必须遵守产品配置中的禁用词要求。

## 微信公众号

定位：品牌内容、产品传播、移动端阅读。

内容目标：让用户快速理解问题、产生兴趣，并愿意继续阅读或点击产品链接。

保留规则：
- 核心痛点
- 产品价值
- 场景化案例
- 清晰的小标题
- 适度的品牌表达

删减规则：
- 过长技术细节
- 大段代码
- 复杂表格
- 太多行业背景铺垫
- 太硬的销售话术

结构规则：
- 标题更有吸引力
- 开头要有场景或痛点
- 段落短，适合手机阅读
- 小标题清晰
- 结尾可以有温和行动引导

语气规则：专业、易读、有传播感，不要像广告硬推。

## 知乎

定位：观点分析、问题回答、经验分享。

内容目标：让文章像是在回答一个真实问题，重点是逻辑、观点和可信度。

保留规则：
- 问题背景
- 逻辑推导
- 对比分析
- 真实痛点
- 客观建议

删减规则：
- 明显营销口吻
- 过多产品自夸
- 口号式表达
- 直接销售转化话术
- 太短、太空的结论

结构规则：
- 标题可以是问题型或观点型
- 开头先提出问题
- 中间分点论证
- 结尾给出判断或建议
- 产品只能自然出现，不能强行推销

语气规则：理性、客观、有观点，减少硬广感。

## CSDN

定位：技术教程、实践说明、开发者学习。

内容目标：让开发者觉得这篇内容有实操价值。

保留规则：
- 技术背景
- 操作步骤
- 功能说明
- 使用场景
- 示例和配置说明
- 可落地的方法

删减规则：
- 过多品牌故事
- 运营化表达
- 情绪化标题
- 空泛价值描述
- 与技术实践无关的段落

结构规则：
- 标题偏实用
- 适合 Markdown
- 可以有步骤编号
- 可以有代码块或配置示例
- 结构清晰，方便搜索

语气规则：技术、实用、清晰，像一篇开发者教程。

## 掘金

定位：开发者经验、技术实践、效率工具分享。

内容目标：让内容像开发者实践总结，不要太官方。

保留规则：
- 实践经验
- 开发者痛点
- 工具效率
- 场景总结
- 方法论

删减规则：
- 太官方的品牌宣传
- 太重的企业营销口吻
- 过长背景铺垫
- 纯概念解释
- 没有实践价值的段落

结构规则：
- 标题有技术价值感
- 开头可以从开发者痛点切入
- 正文偏经验总结
- 语气比 CSDN 轻一点
- 结尾可以有简短总结

语气规则：开发者视角、轻量、实用、经验分享感。

## 后续边界

平台稿生成和平台发布已经分层实现：本文件只定义内容适配规则，发布任务、账号认证和浏览器自动化由 Publisher 模块负责。当前发布侧已支持微信公众号官方 API、掘金非官方接口试点、CSDN 浏览器自动化和知乎浏览器自动化。

内容适配仍需遵守以下边界：

- 不为自动发布编造平台字段、分类、话题或数据。
- 不生成规避审核、诱导互动、刷量或绕过风控的内容。
- 不在平台稿中写入 Cookie、Token、浏览器目录或其他本地敏感配置。
- 发布前细节如话题、标签、摘要、封面、专栏由对应 Publisher 在安全边界内处理。
